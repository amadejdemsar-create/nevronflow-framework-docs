3.0.Decide which tenants to test against do this first

The starter specs default to the demo tenant. That's fine for smoke tests, but real E2E coverage needs tenants where the features actually have data and are enabled. You know which of our tenants have what — Claude doesn't.

What each surface needs

SurfaceRequired tenant state
Admin Features enabled, seeded with real data: orders, QR scans, mood responses, conversations. Demo may be too thin.
PWA Active stays with matching guest emails; catalogue, chat, room service feature flags ON.
TV TV features enabled; a registered test device with a known device token for /api/tv/init/.

Three ways to configure

Option A
One tenant for everything

Override via env vars when running tests:

DEMO_TENANT_SLUG=your-tenant \ DEMO_ADMIN_EMAIL=... \ DEMO_ADMIN_PASSWORD=... \ npm test
Option B
Different tenant per surface

Extend the CREDENTIALS object in fixtures/auth.ts (and create new fixtures for pwa/tv):

export const CREDENTIALS = { demoadmin: { ..., tenant: 'demo' }, managerTest: { ..., tenant: 'x' }, guestTest: { stayId, email, tenant: 'y' }, tvTest: { deviceToken, tenant: 'z' }, };
Option C · recommended long-term
Purpose-built test tenant

Create a new tenant with everything seeded cleanly. Heavier upfront, but gives predictable data every run and isolates E2E from customer tenants.

Don't skip this step

Whichever route you pick, document it in e2e/README.md under a "Test tenants" section — so anyone running the suite later knows which tenant each spec expects and how to reproduce the required state.

3.3.Build tv/ from scratch hardest

TV client at http://localhost:8300. Vue 2, designed for Tizen/LG/STB browsers with memory limits. Remote-control keyboard navigation only — no mouse.

Read first

Read .claude/rules/tv-style.md before writing a single TV test. The constraints (no optional chaining in shipped code, minimal DOM nodes, slow CPUs) affect how your tests should assert and wait.

Specs to create

FileCoverage
tv/fixtures/device-init.tsSimulate TV boot: call /api/tv/init/ with a test device token
tv/boot.spec.ts/init/ returns dashboard; first focusable tile is highlighted
tv/navigation.spec.tsArrow keys move focus through tiles
tv/tv-channels.spec.tsOpen channels, select one, player loads
tv/movies.spec.tsOpen movies module, pick a movie, player loads

Add a tv project to playwright.config.ts

{ name: 'tv-hd', use: { viewport: { width: 1920, height: 1080 }, hasTouch: false, baseURL: 'http://localhost:8300' }, testDir: './tv' }
Key pattern

Use page.keyboard.press('ArrowDown'), not mouse clicks. The TV has no cursor; simulating clicks gives you false positives. Generous timeouts and explicit waits for focus changes matter here more than on the admin or PWA.

Playwright tips
